# 25 Most Used File Trees in Software Development

> Ordered by prevalence of use across the software development ecosystem.
> Last updated: 2026-03-24

| # | Name | Summary Description | Detail |
|---|------|---------------------|--------|
| 1 | **Flat Source Root** | Single directory containing all source files | All `.js`, `.py`, or `.ts` files live in the project root alongside config files. Used in tiny scripts, single-file CLIs, and quick prototypes where structure would be overkill. |
| 2 | **`src/` + `tests/`** | Two-folder separation of source and test code | The most common layout for libraries: production code lives under `src/` and all test files mirror it under `tests/`. Adopted as default by many package managers including `cargo init` (Rust) and `hatch` / `uv` (Python). |
| 3 | **Maven Standard Directory Layout** | Java canonical project layout | Maven defined the de-facto standard for Java projects, separating main and test code by language (`java/`) and resources (`resources/`). Gradle adopted the same layout, making it the universal Java/Kotlin convention. |
| 4 | **Node.js / npm Package** | `src/`, `lib/`, `bin/`, `test/` with `package.json` | Nearly every JavaScript/TypeScript package follows this shape: `src/` for TypeScript source, `lib/` or `dist/` for compiled output, `bin/` for CLI entry points. The `package.json` and `node_modules/` at the root anchor the entire dependency graph. |
| 5 | **Python Package (`src` layout)** | `src/<pkg>/`, `tests/`, `pyproject.toml` | The modern Python packaging standard places the importable package inside `src/` to prevent accidental imports during development. `pyproject.toml` at the root configures build backends and tooling such as mypy, ruff, and pytest. |
| 6 | **React / Create-React-App** | `public/`, `src/`, `src/components/`, `src/pages/` | The canonical frontend SPA layout groups static assets in `public/`, all React code in `src/`, and organises components into named subdirectories. This layout became the baseline that Vite, Next.js and most React starters still reference or extend. |
| 7 | **Next.js App Router** | `app/`, `components/`, `public/`, `lib/` | Next.js 13+ uses the `app/` directory where every folder represents a URL segment and `page.tsx`/`layout.tsx` files define the route UI. Shared UI lives in `components/`, utilities in `lib/`, and `public/` holds static assets served at `/`. |
| 8 | **Monorepo (`apps/` + `packages/`)** | Root workspace containing multiple independent sub-packages | Tools like Turborepo, Nx, and Lerna popularised splitting a codebase into `apps/` (deployable services) and `packages/` (shared libraries), all managed from a single repo root. Each sub-folder is itself a self-contained package with its own manifest file. |
| 9 | **Django Project Layout** | `<project>/`, `<app>/`, `manage.py`, `requirements.txt` | Django separates the outer project config directory (settings, URLs, WSGI) from one or more inner app directories containing `models.py`, `views.py`, `urls.py`, and `migrations/`. `manage.py` at the root serves as the single entry point for all management commands. |
| 10 | **Ruby on Rails MVC** | `app/`, `config/`, `db/`, `lib/`, `test/` | Rails rigidly defines where every file lives: models, views, controllers, helpers, mailers, and jobs all have named subdirectories under `app/`. The strict layout enables Rails' auto-loading and code-generation scaffolds to work without any configuration. |
| 11 | **Go Module Layout** | `cmd/`, `internal/`, `pkg/`, `go.mod` at root | The widely adopted Go project layout places entry-point binaries under `cmd/<n>/main.go`, private packages under `internal/`, and public library packages under `pkg/`. `go.mod` at the root declares the module path and dependency versions. |
| 12 | **Rust Cargo Workspace** | `Cargo.toml` workspace root, `crates/<n>/src/` | Cargo workspaces allow multiple crates to share a `Cargo.lock` and build cache while each living in its own subdirectory with its own `Cargo.toml`. The standard single-crate layout is simply `src/lib.rs` or `src/main.rs` beside the manifest. |
| 13 | **Spring Boot Project** | `src/main/java/<pkg>/`, `src/main/resources/`, `src/test/` | Spring Boot inherits Maven's layout and adds an opinionated resources structure where `application.yml` lives in `src/main/resources/`. Docker and Kubernetes manifests are often stored in a top-level `deploy/` or `k8s/` folder. |
| 14 | **Flutter / Dart Project** | `lib/`, `test/`, `android/`, `ios/`, `web/`, `pubspec.yaml` | Flutter's `flutter create` scaffolds platform-specific directories for every target alongside `lib/` where all Dart application code lives. `pubspec.yaml` at the root declares dependencies, assets, and fonts used across all platforms. |
| 15 | **Android Gradle Project** | `app/src/main/java/`, `app/src/main/res/`, `app/build.gradle` | Android Studio creates a nested structure where Java/Kotlin source, XML resources, and `AndroidManifest.xml` all live under `app/src/main/`. Multiple product flavors or build types can shadow each other using sibling `src/` variant directories. |
| 16 | **iOS / Xcode Project** | `<AppName>/`, `<AppName>Tests/`, `.xcodeproj` | Xcode groups Swift/ObjC source, storyboards, and asset catalogs inside the app target folder, with separate folders for unit and UI test targets. The `.xcodeproj` or `.xcworkspace` bundle at the root is the single entry point for building and signing. |
| 17 | **Infrastructure-as-Code (Terraform)** | `modules/`, `environments/`, `main.tf`, `variables.tf` | Terraform projects split reusable resource definitions into `modules/` and environment-specific configurations into an `environments/` directory. Each directory contains at minimum `main.tf`, `variables.tf`, and `outputs.tf` following HashiCorp's recommended structure. |
| 18 | **Docker / Docker Compose Project** | `Dockerfile`, `docker-compose.yml`, `services/<n>/` | Container-first projects place a `Dockerfile` at the root alongside `docker-compose.yml` for local orchestration. Multi-service repos add a `services/` directory where each subdirectory holds its own `Dockerfile` and source, mirroring a lightweight microservices layout. |
| 19 | **CI/CD Pipeline Layout** | `.github/workflows/`, `Jenkinsfile`, `scripts/`, `deploy/` | Projects with automated pipelines store workflow definitions in vendor-specific paths (`.github/workflows/*.yml` for GitHub Actions, `Jenkinsfile` at root for Jenkins). Shared helper scripts live in `scripts/` and deployment manifests in `deploy/` or `k8s/`. |
| 20 | **Feature-Sliced / Domain-Driven Frontend** | `src/features/<n>/`, `src/shared/`, `src/entities/` | Feature-Sliced Design organises frontend code by business domain rather than technical role, with each feature owning its own components, hooks, store slices, and API calls. A `shared/` layer holds truly cross-cutting utilities and UI primitives used by all features. |
| 21 | **Hexagonal / Clean Architecture** | `domain/`, `application/`, `infrastructure/`, `interfaces/` | Clean Architecture enforces strict layer separation: the innermost `domain/` layer has zero external dependencies, `application/` orchestrates use cases, and `infrastructure/` contains all I/O adapters. This layout makes it straightforward to swap databases or frameworks without touching business logic. |
| 22 | **Jekyll / Hugo Static Site** | `content/` or `_posts/`, `layouts/`, `static/`, `config.toml` | Static site generators expect content in a specific folder (`_posts/` for Jekyll, `content/` for Hugo) and themes in their own directory. Hugo additionally uses `archetypes/` for content templates and `data/` for structured data files that can be queried from templates. |
| 23 | **Data Science / Cookiecutter Data Science** | `data/`, `notebooks/`, `src/`, `models/`, `reports/` | The Cookiecutter Data Science template standardises ML projects with immutable `data/raw/`, processed `data/processed/`, exploratory `notebooks/`, and reusable `src/` pipeline code. `models/` stores serialised artefacts and `reports/figures/` holds generated visualisations for reproducible research. |
| 24 | **Microservices Polyrepo** | One repo per service, each with its own standard layout | Each microservice lives in an independent repository with its own `src/`, `Dockerfile`, CI pipeline, and versioning. Cross-service contracts (OpenAPI specs, protobuf definitions) are often extracted into a shared `contracts/` repository referenced as a Git submodule or package dependency. |
| 25 | **Agent Skills Layout** | `skills/<skill-name>/SKILL.md`, optional `scripts/` and `references/` | Defined by the [agentskills.io](https://agentskills.io/specification) open specification, each skill is a self-contained directory with `SKILL.md` as the single source of truth. The flat `skills/` root enables discovery and version-pinning via commit hashes or checksums. |
